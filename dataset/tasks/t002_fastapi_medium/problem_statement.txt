Bug: response_model_exclude_unset=True leaks fields when response model uses
     Union type with a Pydantic discriminator field

When a FastAPI route uses response_model_exclude_unset=True and the response
model is a Union type resolved via a Pydantic discriminator, unset fields from
the non-matching union branch appear in the serialized response output.

Minimal repro:
    from typing import Union, Literal
    from pydantic import BaseModel
    from fastapi import FastAPI

    class Cat(BaseModel):
        type: Literal["cat"]
        meow: str
        indoor: bool = False      # <-- unset, should be excluded

    class Dog(BaseModel):
        type: Literal["dog"]
        bark: str
        trained: bool = False     # <-- unset, should be excluded

    Animal = Union[Cat, Dog]

    @app.get("/animal", response_model=Animal)
    def get_animal():
        return Cat(type="cat", meow="mrrr")

    # Actual response:   {"type":"cat","meow":"mrrr","indoor":false}
    # Expected response: {"type":"cat","meow":"mrrr"}

Root cause: serialize_response() in fastapi/routing.py resolves the union
model through get_model_definitions() in utils.py, which loses the
exclude_unset flag when traversing the discriminator mapping in the Pydantic
v1/v2 compat layer (_compat.py).

Files to investigate:
- fastapi/routing.py       → serialize_response()
- fastapi/utils.py         → get_model_definitions()
- fastapi/_compat.py       → model_dump() compat shim
- fastapi/openapi/utils.py → ensure schema generation stays consistent
